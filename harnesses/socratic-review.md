# Socratic Review (탁디장)

## Questions

1. 이 변경(또는 추가)의 목표는 무엇인가? (예: `/services` 추가로 고객층 확대)
2. 가장 위험한 부분은 어디인가? (예: 기존 UX 혼란, 폼 필드 변경)
3. 대안은 무엇인가? (예: `/services` 대신 `/` 내 섹션 추가)
4. 되돌리기 어려운 결정은 무엇인가? (예: 폼 필드 추가 → 기존 문의 데이터 스키마 변경)
5. 검증 기준은 무엇인가? (예: `npm run build` 성공, 문의 폼 테스트 pass)

## Verify

- [ ] 5개 질문 모두 답변
- [ ] 대안 1개 이상 검토
- [ ] 결론을 `DECISION-LOG` 기록
- [ ] 이해관계자(디자이너, 클라이언트) 확인 (있으면)

## Example (post-MVP IA 확장)

1. **목표**: "제작+홍보 연계" 서비스 추가로 고객층 넓히기
2. **위험**: 기존 포트폴리오 중심 메시지 분산, 폼 복잡도 증가
3. **대안**: 별도 페이지 대신 `/` 내 "Other Services" 섹션 추가
4. **되돌리기 어려움**: 새 라우트 + 폼 필드 → 후속 마이그레이션 필요
5. **검증**: devfive 벤치마킹 → 설계 → 코드 → 배포 테스트

→ **결정**: `/services`, `/pricing` 신설 (2026-06-15 DECISION-LOG 기록됨)
